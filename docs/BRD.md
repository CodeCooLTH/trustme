# SafePay — Business Requirements Document (BRD)

**เวอร์ชัน:** 1.0
**วันที่:** 23 มีนาคม 2569
**ผู้จัดทำ:** SafePay Team

---

## 1. บทสรุปผู้บริหาร (Executive Summary)

SafePay เป็นแพลตฟอร์ม Escrow Marketplace สำหรับการซื้อขายออนไลน์แบบ C2C (Consumer-to-Consumer) ในประเทศไทย โดยระบบทำหน้าที่เป็น **ตัวกลางพักเงิน** ระหว่างผู้ซื้อและผู้ขาย เพื่อสร้างความมั่นใจและลดความเสี่ยงในการถูกโกงจากการซื้อขายออนไลน์

ระบบจะรับเงินจากผู้ซื้อ พักไว้จนกว่าผู้ซื้อจะได้รับสินค้าและยืนยันความเรียบร้อย จึงปล่อยเงินให้ผู้ขาย หากเกิดปัญหาจะมีระบบข้อพิพาท (Dispute) ให้แอดมินเป็นผู้ตัดสิน

## 2. ปัญหาทางธุรกิจ (Business Problem)

### 2.1 สถานการณ์ปัจจุบัน

- การซื้อขายออนไลน์ในไทยผ่านโซเชียลมีเดีย (Facebook, Instagram, Twitter) มีความเสี่ยงสูง
- ผู้ซื้อไม่มั่นใจว่าจะได้รับสินค้าหลังโอนเงิน
- ผู้ขายไม่มั่นใจว่าผู้ซื้อจะไม่อ้างว่าไม่ได้รับสินค้า
- ไม่มีระบบกลางที่เชื่อถือได้ในการค้ำประกันทั้งสองฝ่าย

### 2.2 ผลกระทบ

- สูญเสียเงินจากการโกง
- ขาดความเชื่อมั่นในตลาดออนไลน์ C2C
- ผู้ขายที่ซื่อสัตย์สูญเสียลูกค้า เพราะผู้ซื้อไม่ไว้ใจ

### 2.3 โอกาส

- ตลาดซื้อขายออนไลน์ C2C ในไทยมีมูลค่าสูงและเติบโตต่อเนื่อง
- ยังไม่มีผู้เล่นหลักที่ให้บริการ Escrow สำหรับ C2C โดยเฉพาะ
- ผู้ใช้พร้อมจ่ายค่าธรรมเนียมเพื่อความปลอดภัย

## 4. ขอบเขตโครงการ (Scope)

### 4.1 ในขอบเขต (In Scope) — MVP

| หมวด | รายละเอียด |
|------|-----------|
| สกุลเงิน | THB (บาทไทย) เท่านั้น |

| บัญชีรับเงิน | 1 บัญชี (แอดมินตั้งค่าได้) |
| ผู้ใช้ | ผู้ซื้อ (Buyer), ผู้ขาย (Seller), แอดมิน (Admin) |

| ข้อพิพาท | แบบง่าย: คืนผู้ซื้อ หรือ ปล่อยผู้ขาย |
| การโอนเงินคืน/ให้ผู้ขาย | แอดมินโอนด้วยมือ |
| ภาษา | ไทยเท่านั้น |
| แจ้งเตือน | In-app เท่านั้น |

- OCR อ่านสลิปอัตโนมัติ
- Webhook ขนส่ง (tracking อัตโนมัติ)
- ระบบ Chat ระหว่าง buyer-seller
- Multi-currency
- Mobile App (iOS/Android)
- LINE Notify / Email notification
- ระบบค่าธรรมเนียม (fee collection)
- API สำหรับ third-party integration
- ระบบ payout อัตโนมัติ

## 6. กระบวนการทางธุรกิจ (Business Process)

### 6.1 Escrow Flow หลัก

```
ผู้ขายสร้างดีล → สร้างลิงก์ส่งให้ผู้ซื้อ
       ↓
ผู้ซื้อเปิดลิงก์ → เข้าสู่ระบบ → ดูรายละเอียด
       ↓
ผู้ซื้อโอนเงินไปบัญชี SafePay → อัพโหลดสลิป
       ↓
แอดมินตรวจสลิป → อนุมัติ/ปฏิเสธ
       ↓
(อนุมัติ) → แจ้งผู้ขาย → ผู้ขายจัดส่ง + ใส่เลข tracking
       ↓
ผู้ซื้อได้รับสินค้า → กดยืนยัน + อัพโหลดรูป
       ↓
ระบบเปลี่ยนสถานะเป็นสำเร็จ → แอดมินโอนเงินให้ผู้ขาย
```

### 6.2 Dispute Flow

```
ผู้ซื้อหรือผู้ขายเปิดข้อพิพาท (ตอนสินค้าถูกจัดส่งแล้ว)
       ↓
แนบหลักฐาน + อธิบายเหตุผล
       ↓
ทั้งสองฝ่ายส่งข้อความ/หลักฐานเพิ่ม
       ↓
แอดมินพิจารณา → ตัดสิน:
  - คืนเงินผู้ซื้อ
  - ปล่อยเงินให้ผู้ขาย
```

## 7. ข้อกำหนดทางธุรกิจ (Business Requirements)

### BR-1: ระบบ Escrow

- **BR-1.1:** ระบบต้องรองรับการพักเงินจากผู้ซื้อจนกว่าจะยืนยันรับสินค้า
- **BR-1.2:** เงินต้องไม่ถูกปล่อยให้ผู้ขายจนกว่าผู้ซื้อจะยืนยัน หรือ deadline หมด
- **BR-1.3:** ต้องมี deadline อัตโนมัติ — ถ้าผู้ซื้อไม่กดยืนยันภายในกำหนด ระบบปล่อยเงินอัตโนมัติ
- **BR-1.4:** Deadline ต้องตั้งค่าได้โดยแอดมิน

### BR-2: ระบบผู้ใช้

- **BR-2.1:** ผู้ใช้ต้องลงทะเบียนผ่าน Facebook Login เท่านั้น (สำหรับยืนยันตัวตน)
- **BR-2.2:** ผู้ซื้อและผู้ขายเป็น role แยกกัน แต่ใช้ Facebook เดียวกันสมัครได้
- **BR-2.3:** ต้องมีระบบ points แสดงความน่าเชื่อถือ

### BR-3: ระบบการชำระเงิน

- **BR-3.1:** รับชำระผ่านโอนธนาคารเท่านั้น (THB)
- **BR-3.2:** แสดงบัญชีรับเงินของ SafePay ให้ผู้ซื้อ
- **BR-3.3:** ผู้ซื้ออัพโหลดสลิปเป็นหลักฐาน
- **BR-3.4:** แอดมินตรวจสอบสลิปด้วยมือ (approve/reject)
- **BR-3.5:** ถ้าสลิปถูก reject — ผู้ซื้อส่งใหม่ได้ สูงสุด 3 ครั้ง
- **BR-3.6:** ป้องกันการส่งสลิปซ้ำ (Idempotency)

### BR-4: ระบบจัดส่ง

- **BR-4.1:** ผู้ขายเลือกขนส่งจากรายชื่อที่แอดมินกำหนด
- **BR-4.2:** ผู้ขายใส่เลข tracking ได้
- **BR-4.3:** ผู้ซื้อติดตามสถานะจัดส่งผ่านลิงก์ได้

### BR-5: ระบบข้อพิพาท

- **BR-5.1:** ผู้ซื้อหรือผู้ขายเปิดข้อพิพาทได้ (เมื่อสถานะเป็น DELIVERED)
- **BR-5.2:** ทั้งสองฝ่ายส่งหลักฐานได้
- **BR-5.3:** แอดมินตัดสิน: คืนผู้ซื้อ หรือ ปล่อยผู้ขาย
- **BR-5.4:** 1 order มีได้เพียง 1 dispute

### BR-6: แอดมิน

- **BR-6.1:** ดู dashboard สถิติรวม (deals, GMV, success rate)
- **BR-6.2:** จัดการผู้ใช้ (ดูรายชื่อ, สถานะ, points)
- **BR-6.3:** ตรวจสอบสลิปการชำระเงิน
- **BR-6.4:** ตัดสินข้อพิพาท
- **BR-6.5:** ตั้งค่าระบบ (บัญชีธนาคาร, deadline, รายชื่อขนส่ง)

## 8. ข้อจำกัดและสมมติฐาน (Constraints & Assumptions)

### ข้อจำกัด

- งบประมาณจำกัด — ใช้ open-source stack ทั้งหมด
- ทีมพัฒนาขนาดเล็ก (1-3 คน)
- ยังไม่มี license ทำธุรกิจการเงิน — เป็น escrow อย่างง่ายก่อน
- ไม่มี API ขนส่ง — tracking เป็น manual

### สมมติฐาน

- ผู้ใช้มี Facebook account
- ผู้ใช้คุ้นเคยกับการโอนเงินผ่านธนาคาร
- แอดมิน 1-2 คน สามารถตรวจสลิปทัน (< 100 deals/เดือน)
- ผู้ใช้ยอมรับ deadline ยืนยันรับสินค้า 3 วัน (ค่าเริ่มต้น)

## 9. ความเสี่ยง (Risks)

| # | ความเสี่ยง | ผลกระทบ | โอกาส | การบรรเทา |
|---|-----------|---------|-------|----------|
| R-1 | ผู้ใช้อัพโหลดสลิปปลอม | สูญเสียเงิน | ปานกลาง | แอดมินตรวจสอบด้วยมือ, อนาคตใช้ OCR |

| R-4 | ระบบล่ม ระหว่างมี deals active | สูญเสียความเชื่อมั่น | ต่ำ | Docker + database backup |
| R-5 | ปัญหากฎหมายเรื่อง escrow | ต้องหยุดให้บริการ | ต่ำ | ปรึกษาทนายก่อน scale |

## 10. แผนระยะ (Roadmap)

### Phase 1: MVP (เดือนที่ 1-2)
- ระบบ Escrow ครบ flow
- Manual payment verification
- Manual payout
- In-app notification

### Phase 2: Improvement (เดือนที่ 3-4)
- OCR สลิปอัตโนมัติ
- LINE Notify
- ระบบ Review/Rating
- ระบบค่าธรรมเนียม

### Phase 3: Scale (เดือนที่ 5-6)
- PromptPay / QR Payment
- API ขนส่ง (auto tracking)
- Payout อัตโนมัติ
- Mobile-responsive PWA

### Phase 4: Expansion (เดือนที่ 7+)
- Mobile App
- Multi-category marketplace
- Chat ระหว่าง buyer-seller
- Multi-currency (SEA expansion)
